# オカレンス図鑑サイト

#  生物多様性情報プラットフォーム 要件定義書 (Draft v0.1)

## 1. プロジェクト概要

- **目的:** 10年、20年先も維持可能な、生物多様性情報の収集・統合・可視化プラットフォームを構築する。
    
- **コアコンセプト:**
    
    - **Cathedral Architecture:** 流行り廃りに左右されない国際標準（W3C/RDF）への準拠。
        
    - **Hard & Soft:** オントロジーによる厳格な意味定義と、ユーザー入力を許容する柔軟性の両立。
        
    - **Aggregation:** 種のページは編集するものではなく、オカレンスデータから自動生成されるものである。
        

## 2. システム構成 (Technology Stack)

- **Frontend:** **Next.js (React)** - 静的エクスポート (`output: 'export'`) を基本とし、将来のデスクトップアプリ化 (Wails) に備える。
    
- **Backend:** **Go (Golang)** - APIサーバー。SPARQLクエリ生成、SHACLバリデーション、検索インデックス同期を担当。
    
- **Database (Main):** **Apache Jena Fuseki** - RDFトリプルストア。生物データ、位置情報 (GeoSPARQL)、オントロジー、SHACLルールを格納。
    
- **Database (Auth):** **PostgreSQL** - ユーザー認証、権限管理、セッション管理。
    
- **Search Engine:** **Meilisearch** - 全文検索、ファセット検索、オントロジー辞書の高速ルックアップ用。
    

## 3. 機能要件

### 3.1 オカレンス管理 (Occurrence)

- **データ登録:**
    
    - 標本記録、観察記録、文献記録を統一的に「オカレンス」として登録できること。
        
    - **Suggestive Input:** 形質（例：夜行性）や環境（例：森林）の入力時、外部オントロジー (PATO, ENVO等) からIDを提案・保管すること。
        
    - **Folksonomy:** 辞書にない用語も一時的に文字列として保存可能とし、後から管理者がオントロジーへ昇格させるフローを持つこと。
        
- **データ検証:**
    
    - 保存時に SHACL を用いてデータの整合性（数値範囲、必須項目）を検証すること。
        
- **詳細表示:**
    
    - 個別のオカレンスについて、地図表示、QRコード生成、修正履歴の表示ができること。
        

### 3.2 種ページ (Taxon Page)

- **自動集計 (Aggregation):**
    
    - ユーザーによる直接編集ではなく、紐付くオカレンスデータから「分布図」「形質統計（サイズ範囲など）」を動的に生成して表示すること。
        
- **コアデータ編集:**
    
    - 学名、シノニム、ホロタイプ指定などの「定義情報」のみ、権限を持つユーザーが編集可能であること。
        

### 3.3 検索機能

- **オカレンス検索:**
    
    - キーワード、分類、期間、採集者による絞り込み。
        
    - 地図範囲指定による検索 (GeoSPARQL)。
        
- **種検索:**
    
    - 分類ツリー（科・属）からのドリルダウン。
        
    - 和名・学名・シノニム（異名）での検索。
        

### 3.4 オントロジー管理

- **用語ブラウザ:** システム内で利用可能な用語（形質・環境など）を検索・閲覧できるページ。
    
- **管理機能:** ユーザー入力された未定義用語（野良タグ）を集計し、正規のオントロジーとして登録するワークフロー。
    

## 4. データモデル・仕様

- **ID体系:** 内部IDには UUID (v4) を使用し、URIとして永続化する。
    
- **採用オントロジー:**
    
    - **DwC (Darwin Core):** 生物記録の基本スキーマ。
        
    - **PATO / RO:** 形質と関係性の記述。
        
    - **Uberon / PO:** 解剖学的構造（動物/植物）。
        
    - **ENVO:** 生息環境。
        
    - **NCBI Taxon:** 生物分類のバックボーン。

## 資料

### 1. 必読の「バイブル」

もし1冊だけ買うなら絶対にこれなのだ。

- **『Semantic Web for the Working Ontologist (3rd Edition)』**
    
    - 著者: Dean Allemang, James Hendler
        
    - **評価:** ★★★★★ (神)
        
    - **理由:** RDFの書き方だけでなく、「なぜそう書くのか？」「どうモデリングすれば推論が効くのか？」という**思考法**が学べる。英語だけど、これを超える本はないのだ。
        
    - ※古い日本語版『実務者のためのセマンティック・ウェブ』もあるけど、情報が古いので頑張って原著の第3版を読むのがおすすめなのだ。
        

### 2. 手を動かして学ぶ系

- **『Learning SPARQL』** (O'Reilly)
    
    - 著者: Bob DuCharme
        
    - **理由:** SPARQLクエリの書き方に特化した本。「こういう検索をしたい時どう書く？」という逆引きとして優秀なのだ。
        

### 3. 日本語の資料（入門用）

- **『セマンティック・ウェブ入門』** (東京電機大学出版局)
    
    - 著者: 川村 隆浩 ほか
        
    - **理由:** 日本の第一人者たちが書いた教科書。基礎理論を日本語で体系的に理解するのに良いのだ。
        
- **Linked Data 関連のSlideShare (Speaker Deck)**
    
    - 日本のLOD (Linked Open Data) チャレンジなどのイベント資料がWebにたくさんあるので、「RDF モデル 設計」などで検索すると、実践的なスライドが見つかるのだ。
        

### 4. Webリファレンス（ブックマーク必須）

- **W3C RDF 1.1 Primer** ([https://www.w3.org/TR/rdf11-primer/](https://www.w3.org/TR/rdf11-primer/))
    
    - 公式の入門書。まずはここをさらっと読むのだ。
        
- **Ontobee** ([http://www.ontobee.org/](https://www.google.com/search?q=http://www.ontobee.org/))
    
    - PATOやUberonのIDを検索するときに毎日使うことになるサイトなのだ。


## フェーズ 0: 土台構築と開通式 (Infrastructure)

**目標:** 全てのミドルウェアが連携し、GoからFusekiに「Hello World」ができる状態にする。

- **Step 1: Docker環境の確立** (1〜2日)
    
    - `docker-compose.yml` を作成し、Fuseki, PostgreSQL, Meilisearch を起動する。
        
    - Fusekiの管理画面でデータセット（`biodb`）を作成し、永続化（再起動してもデータが消えないか）を確認する。
        
- **Step 2: プロジェクト初期化** (1日)
    
    - **Backend (Go):** 標準的なディレクトリ構成（Clean Architecture推奨）を作成。
        
    - **Frontend (Next.js):** `create-next-app` し、`output: 'export'` モードに設定。Tailwind CSS導入。
        
- **Step 3: スパイク（技術検証）** (2〜3日)
    
    - GoからFusekiへ、SPARQLを使って簡単なトリプルを1つ `INSERT` するコードを書く。
        
    - GoからFusekiへ、SPARQLで `SELECT` して結果をコンソールに出す。
        
    - **ここが最初の壁:** ライブラリの選定やRDFの扱いに慣れる期間なのだ。
        

## フェーズ 1: オカレンス登録機能の実装 (The Core)

**目標:** ユーザーがデータを入力し、それがRDFとして構造化されて保存される。**ここが最重要フェーズなのだ。**

- **Step 4: オントロジーインポーターの開発** (3〜5日)
    
    - PATO, ENVO, NCBI Taxon などの定義ファイル（OWL/Turtle）をダウンロード。
        
    - Goでパーサーを書き、必要な「ID」と「ラベル（日/英）」を抽出して **Meilisearch** にインデックスするバッチ処理を作る。
        
    - _※最初は全データじゃなく、テスト用の小規模データでOK。_
        
- **Step 5: 登録UI (Frontend) の実装** (5日)
    
    - Next.jsで登録フォームを作成。
        
    - Meilisearchを検索して、入力候補（Suggest）を出すコンポーネントを作る。
        
    - 「新しい用語（野良タグ）」を入力できるようにする。
        
- **Step 6: 登録ロジック (Backend) の実装** (5日)
    
    - フロントから受け取ったJSONを、Go側で Turtle形式のRDF に変換するロジックを組む。
        
    - SHACLバリデーターを組み込み、最低限の整合性チェック（必須項目など）を行う。
        
    - Fusekiへの書き込み実装。
        

🚀 **マイルストーン 1:** 「タヌキは夜行性」というデータを画面から登録し、Fusekiの管理画面でそのトリプルが確認できる！

## フェーズ 2: データの可視化と種ページ (The Aggregation)

**目標:** 蓄積されたデータを使って、種ごとの「集計ページ」を自動生成する。

- **Step 7: オカレンス詳細ページ** (3日)
    
    - 登録したデータを「閲覧」するページを作る。
        
    - 修正・削除機能の実装。
        
- **Step 8: 集計クエリ (SPARQL) の開発** (5日)
    
    - 「ある種IDを持つオカレンス全ての分布（座標）を取る」クエリ。
        
    - 「ある種IDの体長データの最小・最大・平均を取る」クエリ。
        
    - Fusekiの管理画面でクエリを試行錯誤してチューニングする。
        
- **Step 9: 種ページの実装** (5日)
    
    - Next.jsの動的ルーティング (`/taxon/[id]`) を作成。
        
    - Go API経由で集計データを取得し、グラフや地図（Leaflet/MapLibre）で表示する。
        
    - コアデータ（学名など）の表示。
        

🚀 **マイルストーン 2:** タヌキのオカレンスを10件登録すると、タヌキのページに分布図と「平均体長」が勝手に表示される！

## フェーズ 3: 検索と認証 (Search & Auth)

**目標:** データを「探せる」ようにし、ユーザー管理を導入する。

- **Step 10: ユーザー認証の実装** (3〜5日)
    
    - PostgreSQLにUserテーブルを作成。
        
    - NextAuth.js (Auth.js) と Go API を連携させ、ログイン機能を実装。
        
    - 「誰がデータを登録したか」をRDFに記録 (`dcterms:creator`) するようにロジックを修正。
        
- **Step 11: オカレンス検索の実装** (5日)
    
    - Fusekiに登録されたオカレンスデータを、検索用に **Meilisearch** に同期するバッチ処理を作る。
        
    - 「つくば」「甲虫」などのキーワード検索画面を作る。
        
- **Step 12: 地図検索 (GeoSPARQL) の実装** (オプション/3日)
    
    - 地図上で範囲指定して検索するUIを作る。
        
    - GeoSPARQLクエリの実装。
        

🚀 **マイルストーン 3:** ログインしてデータを登録でき、地図やキーワードで過去の記録を検索できる！＝ **β版リリース**

## フェーズ 4: 運用と進化 (Sustainability)

**目標:** 長期運用のための管理機能と、オントロジーの成長サイクルを作る。

- **Step 13: 用語管理画面**
    
    - ユーザーが勝手に作った「野良タグ」を集計して表示する管理者ダッシュボード。
        
    - 野良タグを正式なオントロジーIDに昇格させる機能。
        
- **Step 14: AI支援機能** (余裕があれば)
    
    - OpenAI APIなどを組み込み、野良タグの定義を自動生成する機能。
        
- **Step 15: デスクトップアプリ化 (Wails)**
    
    - Webアプリが安定したら、Wailsでラップして配布用バイナリを作成。
